Standards & Specifications used in WMB project
Coding Guidelines

If not specified, we use Google Style Guides in WMB project. Most of the time, they're compatible with other coding guidelines e.g. Microsoft standards and PEP. If you have concerns to follow the guides, please feel free to trigger a technical discussion with the team. Thank you.

Frontend / Backend

  • React official practices have higher priority than Google guides.
  • We use Ant Design (antd) as default frontend design language.
  • Google TypeScript Style Guide
  • Google JavaScript Style Guide
  • Google HTML/CSS Style Guide

Runtime / RA

  • Google Python Style Guide
  • Google C++ Style Guide
  • C follows official GNU C or ANSI C standard.
  • Google C# (.NET) Style Guide
  • Google Shell Style Guide
  • Gitlab Go Style Guide

General API

  • RESTful API follows OpenAPI (Swagger) Specification
  • JSON for gRPC follows Google JSON Style Guide
  • gRPC API follows Google Cloud API Design Guide
Global Logging System

Global logging system for WMB project will be based on ElasticSearch (Optional) + Fluentd + Kibana + rsyslog system. It's based on Unified Logging Layer (ULL) concept to provide a standardized logging entrance for all distributed logging sub-systems. It's a cloud native solution with good extensibility.

The system log in Runtime/RA/Vision should generally follow RFC5424 standard, and the logs should be exposed to a global log collector via rsyslog or syslog (including advanced editions of syslog, but please do not use obsoleted libraries).

Internationalization (I18N)

WMB project follows ISO639-ISO3166 standard to define localization languages. It will take effect for both web app and runtime executable binaries and scripts. For POSIX (e.g. Linux) system, the default standard uses underline. But since dash is part of HTML header standard and used by Google, we follows RFC5646 standard to define the language code.

WMB project only supports two languages en-US (default) and zh-CN for the first step.
